# OrionXCore Implementation Notes

This document summarizes the current implementation state of OrionXCore, with emphasis on architecture, request flow, database capabilities, and CLI behavior.

## 1. Current Positioning

OrionXCore is currently an Alpha-to-PoC stage AI agent service with two main usage modes:

- Agent mode via `/v1/agent/respond` and `/v1/agent/stream`
- OpenAI-compatible mode via `/v1/chat/completions`

The strongest implemented vertical today is the ClickHouse database workflow.

## 2. Runtime Architecture

Main runtime pieces:

- API layer: FastAPI routes in `src/orionxcore/api/routes.py`
- Agent orchestration: `src/orionxcore/services/agent.py`
- Model client: `src/orionxcore/llm/openai_compat.py`
- Tool system: `src/orionxcore/tools/*`
- Configuration: `src/orionxcore/config.py`
- CLI client: `src/orionxcore/cli.py`

The app entrypoint is:

- `src/orionxcore/main.py`

## 3. API Surface

### 3.1 Health and discovery

- `GET /health`
- `GET /v1/tools`

### 3.2 Agent mode

- `POST /v1/agent/respond`
- `POST /v1/agent/stream`

This path is server-driven. The service can call tools internally, iterate, and return final output plus intermediate events.

### 3.3 OpenAI-compatible mode

- `POST /v1/chat/completions`

This path is client-driven and designed to behave closer to OpenAI tool-calling semantics:

- supports `tools`
- supports `tool_choice`
- supports `parallel_tool_calls`
- supports `temperature`, `top_p`, `max_tokens`, `stop`
- supports streaming
- supports continuation with `tool` messages
- preserves `reasoning_content` for DeepSeek-style continuation

## 4. Model Integration

The current model client is `OpenAICompatibleClient` in:

- `src/orionxcore/llm/openai_compat.py`

It assumes an OpenAI-compatible `/chat/completions` endpoint and currently:

- sends structured tool definitions
- parses tool calls from returned messages
- flattens content blocks into plain text
- preserves the full raw assistant message in `raw_message`

The preserved `raw_message` is important because some providers, such as DeepSeek, require fields like `reasoning_content` to be returned in later turns.

## 5. Agent Loop

The core server-side loop lives in:

- `src/orionxcore/services/agent.py`

Behavior:

1. Build message list
2. Send request to model
3. If tool calls are returned:
   - emit `tool_call`
   - execute tool
   - emit `tool_result`
   - append tool output as a `tool` message
   - continue iteration
4. If no tool calls are returned:
   - emit final assistant event
   - return response

For database tool results, the service additionally emits fine-grained events:

- `database_trace`
- `database_schema_context`
- `database_sql_attempt`
- `database_result_summary`

These are generated by inspecting the database tool `trace` payload.

## 6. Database Implementation

Database support is intentionally narrowed to ClickHouse.

Implementation file:

- `src/orionxcore/tools/database.py`

### 6.1 Supported operations

- `query`
- `list_tables`
- `describe_table`
- `text_to_sql`

### 6.2 Safety model

Current safety constraints:

- ClickHouse only
- single statement only
- read-only statements only
- row limit enforced
- timeout enforced via `SET max_execution_time`
- optional allowed database whitelist

Config knobs:

- `ORIONXCORE_ENABLE_DATABASE`
- `ORIONXCORE_DATABASE_URL`
- `ORIONXCORE_DATABASE_MAX_ROWS`
- `ORIONXCORE_DATABASE_QUERY_TIMEOUT_SECONDS`
- `ORIONXCORE_DATABASE_ALLOWED_DATABASES`
- `ORIONXCORE_DATABASE_ALLOW_MUTATION`

### 6.3 Schema introspection

Two schema-oriented operations are already available:

- `list_tables`
- `describe_table`

There is also an internal schema context builder used for Text-to-SQL.

### 6.4 Text-to-SQL flow

`text_to_sql` currently does the following:

1. Build schema context from ClickHouse metadata tables
2. Ask the model to generate a read-only ClickHouse SQL query
3. Execute the SQL
4. If execution fails, re-prompt the model with:
   - previous SQL
   - execution error
5. Retry once
6. Return final result plus trace metadata

Returned trace data includes:

- original question
- schema context
- attempt list
- retry reason
- final SQL
- attempt count

## 7. CLI

CLI entrypoint:

- `orionx`

Implementation:

- `src/orionxcore/cli.py`

Current commands:

- `orionx ask "prompt"`
- `orionx ask "prompt" --raw`
- `orionx chat`

### 7.1 `ask`

Sends a single request to `/v1/agent/respond` and formats:

- final assistant response
- iteration events
- tool events
- database trace events

### 7.2 `chat`

Interactive mode:

- auto-generates a `session_id` if omitted
- reuses that `session_id` across turns
- exits on `quit` or `exit`
- supports `--raw`

### 7.3 HTTP behavior

The CLI uses `httpx` with:

- explicit timeout
- `trust_env=False`

This was important because local requests to `127.0.0.1` were being affected by system proxy environment variables.

## 9. Known Gaps

Important gaps still present:

- no persistent session storage yet
- no file-system toolchain yet
- terminal safety is still basic
- no structured auth or rate limiting
- Text-to-SQL result explanation is still minimal
- database support is intentionally limited to ClickHouse for now

## 10. Recommended Next Steps

Best next steps from the current code state:

1. Add result summarization / explanation for database responses
2. Persist `session_id` state server-side
3. Improve CLI interaction controls
4. Add file-system tools for real coding-agent workflows
5. Strengthen terminal approval and sandboxing model
